# Technical Task: Calculator

## Welcome

You have been provided with an existing **codebase** — a simulation of legacy client code.

Scenario: a client handed over a legacy code for you to refactor and improve.

Read the code, research the domain, and understand what problem it solves and how.

---

## What We'd Like You To Do

### Part 1 — Understand the Code

- Read through the codebase and understand what it does
- Research the domain — what is this? What are the rules? Are they correct?
- Be ready to explain it to us as if we've never seen it before

### Part 2 — Write Tests

Write a suite of unit tests for the provided code. You are free to use any test framework you are comfortable with.

Think about:

- Does the code produce the correct results?
- What happens at the boundaries?
- What happens with unexpected inputs?
- What edge cases matter?

> **Hint:** The code may or may not contain bugs. If your tests reveal issues, that's not a problem — that's the point.

### Part 3 — Fix What's Broken

If your tests or your own review uncover any bugs, fix them. Keep track of what you found and how you fixed it.

### Part 4 — Improve It

Make any improvements you think the code needs. There are no wrong answers — we want to see how you think. Some areas you might consider:

- Code quality and structure
- Error handling
- Extensibility
- Adding a **UI** (CLI, web page, or anything you like — your choice)

### Part 5 — Present Your Work

Prepare a short presentation (**10 minutes max**) to walk us through everything. PowerPoint, Google Slides, or any format you prefer.

---

## Your Presentation Should Cover

### 1. What Does This Code Do?
Give us an overview as if we're seeing this for the first time. Explain the domain, the rules, and how the code implements them. Use examples to make it clear — for instance, walk us through what happens when someone enters a specific value. Show us you understand both the code and the real-world problem it solves.

### 2. Your Testing Approach
Tell us about your testing strategy. What framework did you pick and why? What kinds of tests did you write — happy path, edge cases, error cases? Walk us through a few interesting test cases and explain why you chose them. How confident are you that the code is correct after your tests?

### 3. Bugs You Found
Did you find any issues? How did you find them — through tests, code review, or manual testing? Walk us through each bug: what was wrong, what the impact was, and how you fixed it. Before-and-after comparisons are great here.

### 4. Improvements You Made
What did you change and why? Talk us through your reasoning and any trade-offs you considered. If you added a UI, give us a quick demo.

### 5. What You'd Do Next
If you had more time, what would you tackle? Any architectural changes? How would you make this production-ready?

### 6. Q&A
We'll have 10 minutes for questions and open discussion.

---

## What We'll Be Looking For

This isn't just about getting the right answer. Here's what matters to us:

- **Curiosity** — Did you take the time to understand the domain, not just the code?
- **Reasoning** — Can you explain *why* you made the choices you made?
- **Testing mindset** — How do you decide what to test? How do you think about coverage vs. practicality?
- **Communication** — Can you explain technical work clearly? Could a non-developer follow your presentation?
- **Honesty** — If you got stuck or ran out of time, just tell us. We'd rather hear that than a bluff.

---

*Good luck — we look forward to seeing your work!*
